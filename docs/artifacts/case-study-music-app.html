Case Study - The Music App: Origin of the Method

live app - repo - engineering wiki - Command Center - 2026-06-07 to present
"The first app I built letting AI agents do the work, shipping early and often, monitored with the HiTL feedback loop. Born out of my needs as a user. Iterations focused on usability, grounded with directives from me. The first iterations were separate apps per instrument; realizing different instruments were just PROFILES on the same app, we consolidated to a single app on a single source of truth for all components. We used the best available model as ORCHESTRATOR, which designed everything into atomic autonomous swarm sprint plans. HiTL Mission Control HTML pages kept me fully updated; telegram carried the short, direct notification - clear action-required-or-not plus the one link with the specific instructions I needed as the human in the loop."
- the operator's own account of how this app came together. What follows checks each part of it against the repo's own record.

The arc

1. Born from a user's need
2026-06-07first commit
The repo's opening commit already names itself in instrument terms - music/ukulele/ as "Roadcase (uke performance kit)" and music/ukulele/composer/ as "Uke Composer (standalone)," with Guitar noted as next. It carries a Co-Authored-By: Claude Opus 4.8 trailer - an agent wrote code in the app's very first commit, not somewhere down the line.
2. Usability iterations grounded in operator directives
ongoingthe UAT loop
The UAT walkthrough log records 24 numbered findings (U1-U24) written down verbatim as the operator hit them, most turned around the same session (U2: "fretboard section links are all different sizes" fixed same round; U9: a toast that never auto-hid, traced to a shadowed timer variable, extracted into a shared primitive). The Design-System Day pane shows a full day (27 PRs, v88 to v105) spent turning operator directives into a written Element Consistency Law and a 53-assert lint that now enforces it on every PR.
3. The consolidation insight
2026-06-07, 17:31-21:45same day as the first commit
Four hours after the first commit scaffolded separate per-instrument tools, the shape changed: 17:44 extracted a shared, instrument-agnostic songbook out of Roadcase; 17:57 added a Guitar tool reusing that same songbook; 19:48 shipped it as "a versatile app + data tuning-profiles (foundation)"; by 21:45 Roadcase was dropped, and by 01:13 the next morning so was the standalone Uke Composer. What's left today is one profile contract - "profiles are pure data; rendering is shared" - backing 8 instruments (guitar in three tunings, ukulele, banjo, mandolin, mandola, cigar box) off one diagram.js renderer, one chord engine, one source of truth.
4. Orchestrator, HiTL Mission Control, and telegram
standing patternhow the sprints actually ran
The UX sprint-1 plan records it plainly: after a 12-finding adversarial pass, "resolution rulings by mission orchestrator (Opus); chair-level judgment was only required on the undo contract." The atomic queue plan shows the orchestrator's output directly - work stratified into Tier 0 (zero operator decisions, relay-launched as builders free up, capped at 5 concurrent), Tier 1 (one quick pick per item), and Tier 2 (one batched sitting). Five HiTL Mission Control pages carried the operator through each mission in turn: UX personas, wiki + guitarist toolset, blues key/scale mentor, design-system day, and ear-first night. The app's own working notes name the telegram convention directly: verify the preview link is live, make it tappable, and "when the user is on their phone, also send it via the telegram surface" - the short, direct ping the operator describes.

The method

StepWhat happens
CaptureThe operator's raw need or feedback is written down verbatim, not paraphrased - the UAT log and vision-capture docs are the record of this
InterviewA vision or goal-interview document establishes scope before build starts, so agents aren't guessing at intent
Orchestrator IAThe best available model plans the information architecture and resolves adversarial findings, per the sprint-1 plan's "mission orchestrator" ruling
Adversarial gateA second model reviews the design, then the implementation, before it ships - the M-GUIDE fold record is one example, 4 rounds and 10 catches
Swarm wavesWork is tiered by how many operator decisions it needs and relay-launched under a concurrency cap, per the atomic queue plan
HiTL pane + severity pingEach mission gets one HTML Mission Control page; telegram carries the short action-required-or-not ping with the one link that matters
UAT loopThe operator walks the shipped surface and findings turn around the same session, closing back into the next capture
This method extends past this one app: the same capture, orchestrator, adversarial-gate, swarm, HiTL, and UAT loop now organizes engineering work across every active project, not only this one.
