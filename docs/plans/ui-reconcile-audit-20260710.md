# S-UI-RECONCILE Audit - UI Primitive + Keyed-Surface Spelling Inventory (2026-07-10)

Read-only audit of the Music app against the Element Consistency Law ([ssot-registry.md](../../music/engineering-wiki/systems/ssot-registry.md)), [component-conventions.md](../../music/engineering-wiki/ux-philosophy/component-conventions.md), and the regime-B display-spelling contract ([note-spelling.md](../../music/engineering-wiki/theory-engine/note-spelling.md)).
Two classes: (1) keyed surfaces that bypass the display spellers (`dispChord`/`dispChordName`/`Circle.noteInKey`/`preferredTonicName`) and still render retired canonical-sharp names, and (2) rendered elements that diverge from the canonical primitive/token system. Every file:line was read in this pass; nothing here is fixed - fix lanes proposed at the end.

## Root cause of the operator screenshot (A# in an F-major song)

The saved-progression detail view is `renderPractice()` in [music/shared/songbook.js](../../music/shared/songbook.js) (:1438-1615). It renders chord names from the raw transposed `seq` tokens in four places (chip row :1524, sheet chord cards via `renderChordOnly` :467-474, chord row over lyrics via `renderLyricLine` :453-465, transpose readout :1510) - none route through a display speller. The Compose tab's `dispChordName` (:1990) is keyed to the COMPOSE `songKey` closure var, so the song view cannot reuse it as-is; the view already resolves its own key at :1491 (`soloKeyFor` -> `soloKey.key/mode`), which is the correct key source for a view-local disp wrapper. The Stage overlay (`showPerform` :1916-1946) reuses the same sheet renderers plus its own raw key line, so it inherits the identical miss.
